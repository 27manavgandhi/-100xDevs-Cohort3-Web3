## 1. Solidity

**Solidity** is a **programming language** used to create smart contracts for Ethereum and other blockchains that use the Ethereum Virtual Machine (EVM).

### What is a Smart Contract?

A **smart contract** is a self-executing program that runs on the blockchain. It automatically enforces agreements (like sending money or verifying conditions) without needing a middleman.

**Think of it as a digital vending machine:**
- If the conditions are met (insert money, choose item), it executes the result automatically (dispenses item)
- To write our own smart contract, we need to know about Solidity

### Key Characteristics

| Characteristic | Description |
|---|---|
| **Self-executing programs** | Rules written directly into code |
| **Runs on blockchain** | Deployed to Ethereum network |
| **High-level language** | Easy to read and write, similar to JavaScript |
| **Compiled to bytecode** | EVM executes the low-level bytecode |
| **Statically typed** | Variable types must be declared |

## 2. How Solidity Works with EVM

### The Ethereum Virtual Machine (EVM)

The **Ethereum Virtual Machine (EVM)** is conceptually similar to the Java Virtual Machine (JVM) in that both are virtual machines designed to run platform-independent code. Just like JVM differs significantly in their purpose, design, and what they execute.

```
┌─────────────────────────────────────────────────┐
│         ETHEREUM VIRTUAL MACHINE                │
├─────────────────────────────────────────────────┤
│                                                 │
│  Solidity code  →  Bytecode  →  EVM            │
│                                                 │
│  ┌──────────┐     ┌──────────┐   ┌──────────┐ │
│  │ Solidity │ --> │ Bytecode │-->│   EVM    │ │
│  │   Code   │     │(compiled)│   │          │ │
│  └──────────┘     └──────────┘   └──────────┘ │
│                                                 │
│  Runs on: Windows, Ubuntu, Linux, macOS        │
└─────────────────────────────────────────────────┘
```

### Compilation Flow

1. **You write Solidity code** — Human-readable, high-level language
2. **The code is converted into bytecode** — Machine-readable instructions (opcodes)
3. **The bytecode runs on the EVM** — Which works on different systems like Windows, Ubuntu, or Linux

### Key Point

**As a developer, you usually only work with Solidity code.** Rarely do you need to look at the bytecode unless you want to inspect or debug something deep in the blockchain.

## 3. Bytecode and ABI

### What is Bytecode?

**Bytecode** is the **low-level code** that is generated after a smart contract written in a high-level language like Solidity is compiled. It is a sequence of instructions (opcodes) that can be understood and executed by the Ethereum Virtual Machine (EVM).

```
Solidity Code (Chef's Job):
┌─────────────────────────────────────┐
│ contract Sum {                      │
│   function sum(uint a, uint b)      │
│     public pure returns (uint) {    │
│     return a + b;                   │
│   }                                 │
│ }                                   │
└─────────────────────────────────────┘
                ↓
            Compiler
                ↓
Bytecode (Precise Cooking Recipe):
┌─────────────────────────────────────┐
│ 60806040526004361060...            │
│ (This is like a precise cooking     │
│  recipe - the EVM doesn't care      │
│  about how you described the dish)  │
└─────────────────────────────────────┘
```

**Bytecode** is like the **recipe for a dish** that the Ethereum Virtual Machine (EVM) understands and executes.

**Example:** A Solidity contract that adds two numbers

**Real-Life Example:**
- Imagine you're at a robot café. You give the robot a **recipe** (Bytecode), and it follows the exact steps to prepare your coffee
- The chef (EVM) doesn't care about how you described the dish—it only needs the recipe to prepare it

### What is an ABI (Application Binary Interface)?

**ABI** is like the **menu** at a restaurant that explains how to interact with the chef (smart contract).

It tells you:
- The **ingredients** (inputs) you need to give (e.g., `a = 5`, `b = 3`)
- What **dish** (output) you'll get back

**Without the menu (ABI), you won't know how to order your food (or interact with the smart contract).**

```
ABI (Menu):
┌────────────────────────────────────────────┐
│ {                                          │
│   "inputs": [                              │
│     {"name": "a", "type": "uint256"},      │
│     {"name": "b", "type": "uint256"}       │
│   ],                                       │
│   "outputs": [                             │
│     {"name": "", "type": "uint256"}        │
│   ],                                       │
│   "name": "sum",                           │
│   "type": "function"                       │
│ }                                          │
└────────────────────────────────────────────┘
```

### How Bytecode and ABI Work Together (Real-Life Example)

1. **You write a smart contract** (like asking for a dish, e.g., "add two numbers")
   - Example: A Solidity contract that adds two numbers

2. **Bytecode:**
   - This is like a **precise cooking recipe** — The chef (EVM) doesn't care about how you described the dish—it only needs the recipe to prepare it

3. **ABI:**
   - This is the **menu** explaining how you can interact with the chef. It says:
     - **Inputs:** Provide two numbers (`a = 5`, `b = 3`)
     - **Output:** You'll get their sum back

### Example with Solidity Code

**Solidity Code (Chef's Job):**
```solidity
contract Sum {
  function sum(uint a, uint b) public pure returns (uint) {
    return a + b;
  }
}
```

**ABI (Menu):**
```json
{
  "inputs": [
    {"name": "a", "type": "uint256"},
    {"name": "b", "type": "uint256"}
  ],
  "outputs": [{"name": "", "type": "uint256"}],
  "name": "sum",
  "type": "function"
}
```

**User (the user)** look at the menu (ABI) and see the instructions:
- Provide `a = 5` and `b = 3`

**The chef (EVM)** uses the recipe (Bytecode) to calculate the result:
- Output: `5 + 3 = 8`

### Why Are Bytecode and ABI Important?

| Component | Purpose |
|---|---|
| **Bytecode** | Makes sure the smart contract works on the blockchain (The robot chef knows how to execute the instructions) |
| **ABI** | Lets you communicate with the smart contract (You can give inputs and understand the outputs) |

### Summary in Real-Life Terms

- **Bytecode:** Like the recipe for the robot chef—it's how the machine executes instructions
- **ABI:** Like the menu—it tells you how to communicate with the chef (or interact with the smart contract)

## 6. Solidity Version

### Pragma Directive

The **version statement** at the top of a Solidity contract tells the compiler which version of Solidity should be used to compile the contract.

```solidity
pragma solidity ^0.8.0;
```

**Explanation:** This tells the compiler to use any version from **0.8.0** to **0.9.0** (but not 0.9.0 itself)

### Version Compatibility Examples

#### Example 1: Range Version
```solidity
// SPDX-License-Identifier: GPL-3.0
pragma solidity >=0.7.0 <0.9.0;

contract Sum {
  function sum(uint a, uint b) public pure returns (uint) {
    return (a + b, a);
  }
}
```

**Above image shows that this program of solidity require any version between 0.7.0 to 0.9.0**

#### Example 2: Exact Version
```solidity
// SPDX-License-Identifier: GPL-3.0
pragma solidity 0.8.26;

contract Sum {
  function sum(uint a, uint b) public pure returns (uint) {
    return (a + b, a);
  }
}
```

**Above image shows that this program of solidity require only Exact version**

### SPDX License Identifier

```solidity
// SPDX-License-Identifier: GPL-3.0
```

This is a comment that specifies the license for the code. It's a standard way to declare licensing information.

## 8. Variables

In Solidity, **variables** are used to store data within a contract. They can hold different types of values and are fundamental to writing smart contracts.

For example - For a Decentralized Uber app, You would store things like:
- Rides
- Users
- Locations
- Payments in state

## 9. Constructor

A **constructor** in Solidity is a special function that is executed only once during the deployment of the contract. Its primary purpose is to initialize the contract's state variables and set up any required logic when the contract is deployed to the Ethereum blockchain.

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

contract Calculator {
  
  uint256 currentValue;
  
  constructor(uint256 _initialValue) {  // ⚡ Infinite gas (2040 gas)
    currentValue = _initialValue; // Set the initial value
  }
}
```

**Key Points:**
- Runs **only once** when contract is deployed
- Used to initialize state variables
- Can accept parameters
- Costs gas to execute

## 10. Functions

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

contract Calculator {
  
  uint256 currentValue;
  
  constructor(uint256 _initialValue) {
    currentValue = _initialValue;
  }
  
  function add(uint256 value) public {  // ⚡ Infinite gas
    currentValue += value;
  }
}
```

### Function Components

Here:
- `add` is the function name
- `uint256` is type of argument `value`
- `value` name of the argument
- `incrementValue += value` is inside the function body
- `public` shows the visibility

### Visibility Comparison Table

```
┌────────────────────────────────────────────────────────────────┐
│            Visibility Comparison Table                         │
├───────────┬────────────┬─────────────┬──────────┬──────────────┤

│ public    │ ✅         │ ✅          │ ✅       │ ✅           │
│ external  │ ❌(directly)│ ✅         │ ✅       │ ✅           │
│ internal  │ ✅         │ ✅          │ ❌       │ ❌           │
│ private   │ ✅         │ ❌          │ ❌       │ ❌           │
```

## 11. Types of Function

### 1. State-Changing Functions

```
┌────────────────────────────────────────────────────────┐
│           State changing function                      │
├────────────────────────────────────────────────────────┤
│                                                        │
│  contract Calculator {                                 │
│    uint256 public currentValue;                        │
│                                                        │
│    constructor(uint256 _initialValue) {               │
│      currentValue = _initialValue;                     │
│    }                                                   │
│                                                        │
│    function addFunction(uint256 value) public {        │
│      currentValue += value;                            │
│    }                                                   │
│                                                        │
│    function subFunction(uint256 value) public {        │
│      currentValue -= value;                            │
│    }                                                   │
│                                                        │
│    function mulFunction(uint256 value) public {        │
│      currentValue *= value;                            │
│    }                                                   │
│  }                                                     │
└────────────────────────────────────────────────────────┘
```

**These functions change something in the blockchain.**

**Characteristics:**
- They **update variables**, **interact with other contracts**, or **send/receive Ether**
- Requires a **transaction**
- **Costs gas** because it updates the blockchain

**Real-Life Example:**
- Think of it as **making a bank transaction:**
  - If you deposit or withdraw money, the bank updates your account balance (this is a state change)
  - This process takes time and might include a fee (like gas fees on the blockchain)

**Key Point:**
- **Think of a transaction** as the way to interact with a contract when you want to **change or update** something on the blockchain, like transferring money or modifying data

### 2. View Functions

```
┌────────────────────────────────────────────────────────┐
│                  View function                         │
├────────────────────────────────────────────────────────┤
│                                                        │
│  function getValue(uint) public view returns (uint) { │
│    return currentValue;                                │
│  }                                                     │
└────────────────────────────────────────────────────────┘
```

**These functions only read data from the blockchain.**

**Characteristics:**
- They **do not change anything** in the blockchain
- **Real-Life Example:**
  - Imagine you **check your bank balance**:
    - You're just viewing the balance without changing it
    - It's quick, doesn't require a fee, and doesn't affect anything

**Key Point:**
- Can be called **without creating a transaction**
- **No gas fee** when called externally (read-only)

### Summary: State-Changing vs View

| Type | Changes Blockchain | Requires Transaction | Costs Gas | Example |
|---|---|---|---|---|
| **State-Changing** | ✅ Yes | ✅ Yes | ✅ Yes | Making a bank transaction (deposit/withdraw) |
| **View** | ❌ No | ❌ No | ❌ No (external) | Checking your bank balance |

## 12. Inheritance

**Inheritance** allows a contract to inherit the properties and functions of another contract. This enables **code reuse** and the creation of a **hierarchy of contracts**.

```
┌────────────────────────────────────────────────────────┐
│                  Base contract                         │
│         Can be overriden by a child class              │
├────────────────────────────────────────────────────────┤
│                                                        │
│  // SPDX-License-Identifier: MIT                       │
│  pragma solidity ^0.8.0;                               │
│                                                        │
│  contract Vehicle {                                    │
│    string public brand;                                │
│                                                        │
│    constructor(string memory _brand) {                 │
│      brand = _brand;                                   │
│    }                                                   │
│                                                        │
│    function description() public pure virtual          │
│      returns (string memory) {                         │
│      return "This is a vehicle";                       │
│    }                                                   │
│  }                                                     │
└────────────────────────────────────────────────────────┘

┌────────────────────────────────────────────────────────┐
│                  Child contract                        │
├────────────────────────────────────────────────────────┤
│                                                        │
│  import "./Vehicle.sol"; // Import the Vehicle contract│
│                                                        │
│  contract Car is Vehicle {                             │
│                                                        │
│    // Constructor that also initializes parent         │
│    constructor(string memory _brand, uint              │
│      _numberOfDoors) Vehicle(_brand) {                 │
│      numberOfDoors = _numberOfDoors;                   │
│    }                                                   │
│                                                        │
│    // Override the parent function                     │
│    function description() public pure override         │
│      returns (string memory) {                         │
│      return "This is a car";                           │
│    }                                                   │
│  }                                                     │
│                                                        │
│         Child class overrides this fn                  │
└────────────────────────────────────────────────────────┘
```

**Key Concepts:**
- `virtual` keyword: Function can be overridden by child contracts
- `override` keyword: Child contract overrides parent function
- `is` keyword: Declares inheritance
- Constructor chaining: Child calls parent constructor

## 13. If-Else

Conditional logic to execute different code based on conditions.

```solidity
function isEven(uint number) public pure returns (string memory) {
  if (number % 2 == 0) {
    return "Even";
  } else {
    return "Odd";
  }
}
```

**Explanation:** Checks if a number is divisible by 2. If true, returns "Even," otherwise "Odd."

## 14. Loops

Used to repeat code multiple times.

```solidity
function oddNumbers(uint limit) public {
  for (uint i = 0; i < limit; i++) {
    numbers.push(i); // Adds numbers 0 to (limit-1) to the array
  }
}
```

**Explanation:** A `for` loop iterates from `0` to `limit-1` and adds each value to an array.

## 15. Require

Ensures conditions are met; otherwise, it stops execution.

```solidity
function deposit(uint amount) public {
  require(amount > 0, "Amount must be greater than 0");
  balance[msg.sender] += amount;
}
```

**Explanation:** Ensures the deposit amount is greater than 0 before updating the balance.

---

## 16. Mappings

A key-value store, like a dictionary.

```solidity
mapping(address => uint) public balances;
```

**Explanation:** Maps each wallet address (`address`) to a balance (`uint`).

---

## 17. Arrays

Lists to store multiple items.

```solidity
uint[] public numbers;

function addNumber(uint num) public {
  numbers.push(num); // Adds a number to the array
}
```

**Explanation:** Stores a list of numbers, and the `push` function adds a new number.

---

## 18. Structs

Custom data types to group multiple variables.

```solidity
struct Person {
  string name;
  uint age;
}

Person public person = Person("Alice", 25);
```

**Explanation:** Groups `name` and `age` into a single custom data type called `Person`. The contract initializes `person` with "Alice" and age 25.
