## 1. What is GitOps?

GitOps is a modern approach to managing infrastructure and application deployment using **Git as the single source of truth**. It applies DevOps practices—like version control, CI/CD, and automation—to infrastructure.

### Real-Life Analogy

Think of GitOps like a **blueprint-driven construction project**:

- **Traditional Approach**: Workers manually build based on verbal instructions. If someone makes a mistake, there's no record of what should have been done.
- **GitOps Approach**: All construction blueprints (Git repo) are centrally stored. An automated system constantly checks if the building matches the blueprint and automatically fixes any deviations.

```
┌─────────────────────────────────────────────────┐
│              GitOps Workflow                     │
├─────────────────────────────────────────────────┤
│                                                  │
│  Developer           Git Repository             │
│     │                     │                      │
│     │ 1. Push changes     │                      │
│     ├────────────────────>│                      │
│     │                     │                      │
│     │                     │ 2. GitOps Agent      │
│     │                     │    watches repo      │
│     │                     │<────────┐            │
│     │                     │         │            │
│     │                     │ 3. Pull │            │
│     │                     │    changes           │
│     │                     │         │            │
│     │              Kubernetes Cluster            │
│     │                     │         │            │
│     │                     │ 4. Apply to cluster  │
│     │                     ├────────>│            │
│     │                     │         │            │
│     │                     │ 5. Continuous        │
│     │                     │    reconciliation    │
│     │                     │<────────┘            │
│                                                  │
└─────────────────────────────────────────────────┘
```

### Core Concepts of GitOps

1. **Git as the Source of Truth**:
   - All infrastructure and deployment configurations are stored in Git repositories
   - Kubernetes manifests, Helm charts, Terraform files, etc.

2. **Declarative Descriptions**:
   - Desired system state is described in a declarative way (e.g., YAML files for Kubernetes)
   - Not imperative scripts ("do this, then do that") but declarative configs ("this is what I want")

3. **Automated Reconciliation**:
   - A GitOps agent (like Argo CD or Flux) continuously watches the Git repo
   - Automatically applies changes to match the desired state
   - Detects and corrects any drift from the desired state

4. **Pull-Based Deployment**:
   - Instead of pushing changes from CI/CD to production
   - The production cluster pulls changes from Git

5. **Audit and Rollback**:
   - Every change is tracked via Git commits, enabling easy auditing and rollback by reverting commits

## 2. Why GitOps?

GitOps provides several advantages over traditional CI/CD approaches:

### Benefits Table

| Benefit | Description |
|---------|-------------|
| **Version Control** | All infrastructure changes are versioned in Git, providing complete history |
| **Auditability** | Every change has a commit message, author, and timestamp |
| **Easy Rollback** | Revert to any previous state by reverting Git commits |
| **Security** | Cluster credentials don't leave the cluster; no external CI/CD access needed |
| **Declarative** | Describe what you want, not how to get there |
| **Self-Healing** | Automatic drift detection and correction |
| **Collaboration** | Use Git workflows (PRs, code review) for infrastructure changes |
| **Disaster Recovery** | Entire cluster state is in Git; easy to recreate |

### Traditional CI/CD vs GitOps

```
TRADITIONAL CI/CD (Push-Based):
┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│   Git Repo  │────>│  CI/CD Tool │────>│   Cluster   │
│             │     │  (Jenkins,  │     │             │
│  (Code)     │     │   GitLab)   │     │  (Target)   │
└─────────────┘     └─────────────┘     └─────────────┘
                          │
                          │ Needs cluster credentials
                          │ Security risk
                          └─> External access to production


GITOPS (Pull-Based):
┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│   Git Repo  │<────│ ArgoCD/Flux │<────│   Cluster   │
│             │     │  (In-Cluster│     │             │
│ (Manifests) │     │   Agent)    │     │  (Target)   │
└─────────────┘     └─────────────┘     └─────────────┘
                          │
                          │ No external credentials needed
                          │ More secure
                          └─> Cluster pulls its own config
```

### Why Choose GitOps?

1. **Single Source of Truth**: Git repo contains everything needed to recreate your infrastructure
2. **Better Security**: No need to expose cluster credentials to external CI/CD systems
3. **Improved Developer Experience**: Use familiar Git workflows for infrastructure
4. **Consistency**: Same deployment process across all environments (dev, staging, prod)
5. **Compliance**: Complete audit trail of who changed what and when

## 3. ArgoCD — Introduction

**Argo CD** (short for *Argo Continuous Delivery*) is a **GitOps tool for Kubernetes** that continuously deploys and manages applications based on the state defined in a Git repository.

### Real-Life Analogy

ArgoCD is like a **home automation system** that:
- Has a master control app (your Git repo) that defines how every room should be
- Constantly monitors your home
- Automatically adjusts lights, temperature, and devices to match your settings
- If someone manually changes something, it automatically corrects it back

### Key Features

| Feature | Description |
|---------|-------------|
| **Automated Sync** | Automatically deploys changes from Git to Kubernetes |
| **Self-Healing** | Detects and corrects configuration drift |
| **Web UI** | Beautiful dashboard to visualize applications and their health |
| **Rollback** | Easy rollback to any previous Git commit |
| **Health Status** | Monitors application health and resource status |
| **SSO Integration** | Supports OIDC, OAuth, LDAP, SAML |
| **Multi-Cluster** | Manage applications across multiple clusters |
| **Webhook Support** | Trigger syncs via Git webhooks |

### ArgoCD Components Explained

1. **API Server**:
   - The front-facing component
   - Exposes REST API and serves the Web UI
   - Handles authentication and authorization

2. **Repository Server**:
   - Internal service that maintains a local cache of Git repos
   - Responsible for generating Kubernetes manifests from various sources (raw YAML, Helm, Kustomize, etc.)

3. **Application Controller**:
   - The core reconciliation engine
   - Continuously monitors Git repositories
   - Compares desired state (Git) with actual state (cluster)
   - Triggers synchronization when differences are detected

4. **Dex (Optional)**:
   - Identity service for SSO integration
   - Connects to external identity providers (GitHub, GitLab, LDAP, etc.)

5. **Redis**:
   - Used for caching to reduce load on Git repos and Kubernetes API
   - Stores application state and session information

## 4. Installing ArgoCD

ArgoCD can be installed in multiple ways. Let's explore the most common approaches.

### Approach #1: Install via kubectl (Recommended)

This is the official and most straightforward method.

**Step 1: Create namespace**

```bash
kubectl create namespace argocd
```

**Step 2: Apply ArgoCD manifests**

```bash
kubectl apply -n argocd -f https://raw.githubusercontent.com/argoproj/argo-cd/stable/manifests/install.yaml
```

This installs:
- All ArgoCD components (API server, repo server, application controller, etc.)
- Required RBAC roles and service accounts
- CRDs (Custom Resource Definitions) for Applications, AppProjects, etc.

**Step 3: Verify installation**

```bash
kubectl get pods -n argocd
```

You should see pods like:
```
argocd-application-controller-0
argocd-repo-server-xxx
argocd-server-xxx
argocd-redis-xxx
argocd-dex-server-xxx
```

Wait until all pods are in `Running` state.

### Approach #2: Install from DigitalOcean Marketplace

For managed Kubernetes clusters (like DigitalOcean), you can install from their marketplace:

1. Go to DigitalOcean Marketplace
2. Search for "ArgoCD"
3. Click "Install"
4. Select your Kubernetes cluster
5. DigitalOcean will handle the installation

This provides a one-click installation with reasonable defaults.

### Approach #3: Install ArgoCD CLI (Not required but recommended)

The ArgoCD CLI is useful for managing ArgoCD from the command line.

**macOS (using Homebrew):**

```bash
brew install argocd
```

**Linux:**

```bash
brew install argocd
```

Or download directly:

```bash
curl -sSL -o /usr/local/bin/argocd https://github.com/argoproj/argo-cd/releases/latest/download/argocd-linux-amd64
chmod +x /usr/local/bin/argocd
```

**Windows:**

Download the latest `.exe` from the [ArgoCD releases page](https://github.com/argoproj/argo-cd/releases).

## 5. Accessing the ArgoCD UI

Once installed, you need to access the ArgoCD UI. The service is not exposed externally by default.

### Method 1: Port Forwarding (Quick & Easy)

Forward the ArgoCD server port to your local machine:

```bash
kubectl port-forward svc/argocd-server -n argocd 8080:443
```

Now access ArgoCD at: **http://localhost:8080**

### Method 2: Expose via LoadBalancer (Production)

Change the `argocd-server` service type to LoadBalancer:

```bash
kubectl patch svc argocd-server -n argocd -p '{"spec": {"type": "LoadBalancer"}}'
```

Get the external IP:

```bash
kubectl get svc argocd-server -n argocd
```

Access ArgoCD at the external IP.

### Method 3: Ingress (Most Common in Production)

Create an Ingress resource to expose ArgoCD:

```yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: argocd-server-ingress
  namespace: argocd
  annotations:
    cert-manager.io/cluster-issuer: letsencrypt-prod
spec:
  rules:
  - host: argocd.example.com
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: argocd-server
            port:
              number: 443
  tls:
  - hosts:
    - argocd.example.com
    secretName: argocd-tls
```

## 6. Getting the Initial Password

ArgoCD automatically generates an initial admin password during installation.

### Get the Password

```bash
kubectl -n argocd get secret argocd-initial-admin-secret -o jsonpath="{.data.password}" | base64 -d; echo
```

This decodes the base64-encoded password and prints it.

**Example output:**

```
jksqnO7H3NKL9qhP
```

### Login Credentials

- **Username**: `admin`
- **Password**: The output from the command above

### Change the Password (Recommended)

After first login, change the password:

**Via UI:**
1. Click on "User Info" in the top-right
2. Select "Update Password"
3. Enter old and new passwords

**Via CLI:**

```bash
argocd login localhost:8080
argocd account update-password
```

### Delete Initial Secret (After Password Change)

For security, delete the initial secret after changing the password:

```bash
kubectl -n argocd delete secret argocd-initial-admin-secret
```

## 7. Creating Your First Application in ArgoCD

Applications in ArgoCD are defined declaratively, usually as YAML manifests.

### Application Components

An ArgoCD Application has three main parts:

1. **Source**: Where the manifests are stored (Git repo, Helm chart, etc.)
2. **Destination**: Where to deploy (cluster and namespace)
3. **Sync Policy**: How to handle synchronization (manual, automatic, self-heal, etc.)

### Example Application Manifest

```yaml
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: guestbook
  namespace: argocd
spec:
  # Source: Git repository containing manifests
  source:
    repoURL: https://github.com/argoproj/argocd-example-apps.git
    targetRevision: HEAD
    path: guestbook
  
  # Destination: Target cluster and namespace
  destination:
    server: https://kubernetes.default.svc
    namespace: default
  
  # Sync policy: How to handle syncing
  syncPolicy:
    automated:
      prune: true       # Delete resources not in Git
      selfHeal: true    # Auto-sync if cluster state drifts
    syncOptions:
    - CreateNamespace=true
```

### Apply the Application

```bash
kubectl apply -f application.yaml
```

### View in UI

Go to the ArgoCD UI and you'll see your application. It shows:
- Application health (Healthy, Degraded, Progressing, etc.)
- Sync status (Synced, OutOfSync)
- All Kubernetes resources managed by this application

## 8. Sync Strategies

ArgoCD supports different synchronization strategies.

### Manual Sync

The application will **not** automatically sync. You must manually trigger syncs via:

- UI: Click "Sync" button
- CLI: `argocd app sync <app-name>`

**Configuration:**

```yaml
syncPolicy:
  # No automated section = manual sync
```

### Automatic Sync

ArgoCD automatically syncs when it detects changes in Git.

**Configuration:**

```yaml
syncPolicy:
  automated: {}
```

### Automatic Sync with Prune

Automatically deletes resources that exist in the cluster but not in Git.

**Configuration:**

```yaml
syncPolicy:
  automated:
    prune: true
```

### Automatic Sync with Self-Heal

Automatically corrects any manual changes made to the cluster (drift detection).

**Configuration:**

```yaml
syncPolicy:
  automated:
    selfHeal: true
```

### Sync Policy Comparison

| Policy | Auto-Sync | Prune | Self-Heal | Use Case |
|--------|-----------|-------|-----------|----------|
| Manual | ❌ | ❌ | ❌ | Development, careful control |
| Auto | ✅ | ❌ | ❌ | Basic automation |
| Auto + Prune | ✅ | ✅ | ❌ | Clean deployments |
| Auto + Self-Heal | ✅ | ❌ | ✅ | Prevent manual drift |
| Auto + Prune + Self-Heal | ✅ | ✅ | ✅ | Full automation (production) |

## 9. Self-Healing in ArgoCD

**Self-healing** is one of the most powerful features of ArgoCD. It automatically corrects any **drift** between the desired state (Git) and the actual state (cluster).

### What is Drift?

Drift occurs when someone manually changes resources in the cluster, making them different from what's defined in Git.

**Example:**
- Git says: `replicas: 3`
- Someone runs: `kubectl scale deployment my-app --replicas=5`
- Result: Drift detected!

### How Self-Healing Works

With self-healing enabled, ArgoCD:

1. Continuously monitors the cluster
2. Detects any differences from Git
3. Automatically re-applies the Git state
4. Logs the correction

### Enable Self-Healing

```yaml
syncPolicy:
  automated:
    selfHeal: true
```

### Self-Healing in Action

**Scenario:**

```
1. Git defines: replicas: 3
2. ArgoCD deploys: 3 replicas running
3. Admin manually scales: kubectl scale ... --replicas=5
4. ArgoCD detects drift
5. ArgoCD automatically reverts to 3 replicas
6. Git remains the source of truth
```

### When to Use Self-Healing

| Scenario | Self-Heal? |
|----------|-----------|
| Production (strict control) | ✅ Yes |
| Development (need manual testing) | ❌ No |
| Staging (testing deployments) | ✅ Yes |
| Demo environments | ✅ Yes |

## 10. Pull-Based vs Push-Based Deployment

Understanding the deployment model is key to understanding GitOps.

### Push-Based Deployment (Traditional CI/CD)

```
┌──────────┐      ┌──────────┐      ┌──────────┐
│  Git     │─────>│  CI/CD   │─────>│ Cluster  │
│  Commit  │      │  (Push)  │      │          │
└──────────┘      └──────────┘      └──────────┘
                       │
                       └─> Requires cluster credentials
                           External access to production
```

**Characteristics:**
- CI/CD tool (Jenkins, GitLab CI, GitHub Actions) **pushes** changes to the cluster
- Requires cluster credentials in CI/CD system
- External system has write access to production
- Security risk if CI/CD is compromised

### Pull-Based Deployment (GitOps with ArgoCD)

```
┌──────────┐      ┌──────────┐      ┌──────────┐
│  Git     │<─────│  ArgoCD  │<─────│ Cluster  │
│  Repo    │      │ (In-Clstr│      │          │
└──────────┘      └──────────┘      └──────────┘
                       │
                       └─> ArgoCD runs inside cluster
                           No external credentials needed
```

**Characteristics:**
- ArgoCD runs **inside** the cluster
- ArgoCD **pulls** changes from Git
- No need to expose cluster credentials externally
- More secure architecture

### Why Pull-Based is Better

| Aspect | Push-Based | Pull-Based |
|--------|------------|------------|
| Security | CI/CD needs cluster creds | Creds stay in cluster |
| Audit | CI/CD logs | Git commits |
| Rollback | Re-run pipeline | Git revert |
| Drift Detection | None | Automatic |
| Self-Healing | Manual | Automatic |
| Disaster Recovery | Difficult | Easy (just point to Git) |

## 11. GitOps Best Practices

### Repository Structure

**Option 1: Mono-Repo (All in one repository)**

```
my-app/
├── src/                    # Application code
│   ├── index.js
│   └── ...
├── manifests/              # Kubernetes manifests
│   ├── deployment.yaml
│   ├── service.yaml
│   └── ingress.yaml
└── Dockerfile
```

**Option 2: Separate Repos (Recommended for larger teams)**

```
App Code Repo:              Config Repo:
my-app/                     my-app-config/
├── src/                    ├── base/
│   ├── index.js            │   ├── deployment.yaml
│   └── ...                 │   ├── service.yaml
└── Dockerfile              │   └── kustomization.yaml
                            ├── overlays/
                            │   ├── dev/
                            │   ├── staging/
                            │   └── prod/
```

**Why separate repos?**
- Different access controls (developers vs ops)
- Independent change tracking
- Clearer separation of concerns
- Easier to manage multiple environments

### Environment Management

Use **branches** or **directories** for environments:

**Branch-Based:**
```
main           # Production
├── staging    # Staging
└── dev        # Development
```

**Directory-Based (Kustomize):**
```
config-repo/
├── base/
│   ├── deployment.yaml
│   └── service.yaml
└── overlays/
    ├── dev/
    ├── staging/
    └── prod/
```

### Workflow Pattern

```
1. Developer commits code
   ↓
2. CI builds and tests
   ↓
3. CI builds Docker image
   ↓
4. CI updates image tag in Git config repo
   ↓
5. ArgoCD detects change in Git
   ↓
6. ArgoCD syncs to cluster
   ↓
7. Application updated
```

### Security Best Practices

1. **Use SSH keys** for private repositories (not HTTPS with passwords)
2. **Limit ArgoCD permissions** using RBAC
3. **Enable SSO** for user authentication
4. **Use separate repos** for app code and configs
5. **Encrypt secrets** (use Sealed Secrets or external secret managers)
6. **Audit Git commits** regularly
7. **Use signed commits** for critical changes

## 12. Debugging ArgoCD

### Check Application Status

**Via UI:**
- Go to ArgoCD dashboard
- Click on your application
- View resource tree and events

**Via CLI:**

```bash
# Get application status
argocd app get <app-name>

# Get detailed status
argocd app get <app-name> --show-operation

# Watch sync progress
argocd app wait <app-name>
```

### View Logs

```bash
# Application Controller logs
kubectl logs -n argocd deployment/argocd-application-controller

# API Server logs
kubectl logs -n argocd deployment/argocd-server

# Repo Server logs
kubectl logs -n argocd deployment/argocd-repo-server
```

### Common Issues

| Issue | Cause | Solution |
|-------|-------|----------|
| App stuck "Progressing" | Resource not becoming healthy | Check resource events, logs |
| Sync fails | Invalid manifests | Validate YAML syntax |
| "OutOfSync" constantly | Self-heal disabled, manual changes | Enable self-heal or revert manual changes |
| Can't access UI | Port forward not running | Run `kubectl port-forward` again |
| Authentication failed | Wrong credentials | Get password via `kubectl get secret` |
| Slow sync | Large repo, no caching | Enable Redis caching |

### Health Checks

ArgoCD determines resource health based on:

- **Deployments**: All replicas ready
- **StatefulSets**: All replicas ready
- **DaemonSets**: All desired pods scheduled
- **Services**: Endpoints ready
- **Pods**: Running and ready
- **Jobs**: Completed successfully

3. **Pull-Based Deployment**: More secure than push-based CI/CD
4. **Self-Healing**: Automatic drift detection and correction
5. **Auditability**: Every change tracked via Git commits
6. **Easy Rollback**: Just revert a Git commit
7. **Beautiful UI**: Visualize application state and health

**ArgoCD transforms Kubernetes deployments from manual, error-prone processes into automated, auditable, and reliable workflows.**

| GitOps | Using Git as the single source of truth for declarative infrastructure |
| ArgoCD | GitOps continuous delivery tool for Kubernetes |
| Declarative | Describing desired state, not imperative commands |
| Reconciliation | Process of making actual state match desired state |
| Pull-Based | Cluster pulls config from Git (vs CI/CD pushing to cluster) |
| Self-Healing | Automatic correction of drift from desired state |
| Drift | Difference between actual cluster state and Git state |
| Sync | Applying Git state to cluster |
| Application | ArgoCD resource representing a deployed app |
| Source | Git repo location of manifests |
| Destination | Target cluster and namespace |
| Prune | Deleting resources not in Git |
| Application Controller | Core ArgoCD component that does reconciliation |
| Repository Server | Component that fetches from Git and generates manifests |
| API Server | Component that serves UI and REST API |
