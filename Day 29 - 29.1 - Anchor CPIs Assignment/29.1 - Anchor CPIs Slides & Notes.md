## 1. What are CPIs (Cross-Program Invocations)?

**Cross-Program Invocation (CPI)** is when one Solana program calls another program's instruction. This is Solana's composability mechanism—similar to calling external contracts in Ethereum.

### Real-Life Analogy

Think of CPIs like **customer service transfers**:

- **Without CPI**: Customer calls bank, bank says "sorry, you need to call the insurance department separately"
- **With CPI**: Customer calls bank, bank internally transfers you to insurance while staying on the same call

```
┌────────────────────────────────────────────────────┐
│              CPI Flow Diagram                       │
├────────────────────────────────────────────────────┤
│                                                     │
│  User Transaction                                  │
│       │                                            │
│       ▼                                            │
│  ┌─────────────────┐                              │
│  │   Program A     │                              │
│  │  (Your code)    │                              │
│  │                 │                              │
│  │  fn transfer()  │                              │
│  │     ├─────────────────────┐                    │
│  │     │ CPI Call            │                    │
│  │     │                     ▼                    │
│  │     │           ┌──────────────────┐          │
│  │     │           │   Program B      │          │
│  │     │           │ (System Program) │          │
│  │     │           │                  │          │
│  │     │           │  create_account  │          │
│  │     │           └──────────────────┘          │
│  │     │                     │                    │
│  │     ◄─────────────────────┘                    │
│  │  return result                                 │
│  └─────────────────┘                              │
│                                                     │
└────────────────────────────────────────────────────┘
```

### Why Use CPIs?

| Reason | Description |
|--------|-------------|
| **Composability** | Build on existing programs (like DeFi composability) |
| **Code Reuse** | Don't reimplement System Program functionality |
| **Modularity** | Separate concerns into different programs |
| **Interoperability** | Your program can interact with any other program |
| **Efficiency** | Leverage battle-tested programs (SPL Token, System) |

### Common CPI Use Cases

1. **Creating Accounts**: Call System Program to create new accounts
2. **Token Operations**: Call SPL Token program to mint/burn/transfer
3. **DeFi Interactions**: Call Serum DEX, Raydium, etc.
4. **Oracle Data**: Call Chainlink or Pyth for price feeds
5. **NFT Operations**: Call Metaplex for NFT minting

## 4. Writing Native Contract with CPI

### Native Solana CPI Example

```rust
use borsh::{BorshDeserialize, BorshSerialize};
use solana_program::{
    entrypoint,
    account_info::{next_account_info, AccountInfo},
    entrypoint::ProgramResult,
    pubkey::Pubkey,
    program_error::ProgramError,
    msg,
    rent::Rent,
    system_instruction,
    program::invoke,
};

#[derive(BorshSerialize, BorshDeserialize)]
struct CounterState {
    count: u32,
}

#[derive(BorshSerialize, BorshDeserialize)]
enum CounterInstruction {
    Initialize,
    Double,
    Half,
}

entrypoint!(process_instruction);

fn process_instruction(
    program_id: &Pubkey,
    accounts: &[AccountInfo],
    instruction_data: &[u8]
) -> ProgramResult {
    let instruction = CounterInstruction::try_from_slice(instruction_data)?;
    
    match instruction {
        CounterInstruction::Initialize => {
            msg!("Initializing counter");
            
            let mut iter = accounts.iter();
            let data_account = next_account_info(&mut iter)?;
            let payer = next_account_info(&mut iter)?;
            let system_program = next_account_info(&mut iter)?;
            
            // Check if payer is signer
            if !payer.is_signer {
                return Err(ProgramError::MissingRequiredSignature);
            }
            
            // Calculate space needed for CounterState
            let space = 4; // u32 = 4 bytes
            
            // Calculate rent exemption amount
            let rent = Rent::get()?;
            let lamports = rent.minimum_balance(space);
            
            // Create the account using CPI to System Program
            let create_account_ix = system_instruction::create_account(
                payer.key,           // Payer of rent
                data_account.key,    // New account address
                lamports,            // Lamports for rent exemption
                space as u64,        // Space to allocate
                program_id,          // Owner of new account
            );
            
            // Invoke the System Program
            invoke(
                &create_account_ix,
                &[
                    payer.clone(),
                    data_account.clone(),
                    system_program.clone(),
                ],
            )?;
            
            // Initialize the account data
            let counter_state = CounterState { count: 1 };
            counter_state.serialize(&mut *data_account.data.borrow_mut())?;
            
            msg!("Counter initialized");
        },
        
        CounterInstruction::Double => {
            msg!("Doubling counter");
            
            let mut iter = accounts.iter();
            let data_account = next_account_info(&mut iter)?;
            
            // Check if the account is owned by this program
            if data_account.owner != program_id {
                return Err(ProgramError::IncorrectProgramId);
            }
            
            let mut counter_state = CounterState::try_from_slice(&data_account.data.borrow())?;
            counter_state.count = counter_state.count * 2;
            counter_state.serialize(&mut *data_account.data.borrow_mut())?;
        },
        
        CounterInstruction::Half => {
            msg!("Halving counter");
            
            let mut iter = accounts.iter();
            let data_account = next_account_info(&mut iter)?;
            
            if data_account.owner != program_id {
                return Err(ProgramError::IncorrectProgramId);
            }
            
            let mut counter_state = CounterState::try_from_slice(&data_account.data.borrow())?;
            counter_state.count = counter_state.count / 2;
            counter_state.serialize(&mut *data_account.data.borrow_mut())?;
        },
    }
    
    Ok(())
}
```

### Key Points in Native CPI

1. **system_instruction::create_account()**: Creates the instruction
2. **invoke()**: Executes the CPI
3. **Account order matters**: Payer, new account, system program
4. **Rent calculation**: Must provide enough lamports for rent exemption

## 5. Writing Anchor Contract with CPI

Anchor makes CPIs dramatically simpler with helper functions.

### Anchor CPI Approach #1: Using `anchor_lang::system_program`

```rust
use anchor_lang::prelude::*;
use anchor_lang::system_program::{transfer, Transfer};

declare_id!("YourProgramIDHere");

#[program]
pub mod cpi_contract {
    use super::*;

    pub fn sol_transfer(ctx: Context<SolTransfer>, amount: u64) -> Result<()> {
        let from_pubkey = ctx.accounts.sender.to_account_info();
        let to_pubkey = ctx.accounts.recipient.to_account_info();
        let program_id = ctx.accounts.system_program.to_account_info();
        
        // Create CPI context
        let cpi_context = CpiContext::new(
            program_id,
            Transfer {
                from: from_pubkey,
                to: to_pubkey,
            },
        );
        
        // Execute the CPI
        transfer(cpi_context, amount)?;
        
        Ok(())
    }
}

#[derive(Accounts)]
pub struct SolTransfer<'info> {
    #[account(mut)]
    pub sender: Signer<'info>,
    
    #[account(mut)]
    /// CHECK: This account is safe because we're just transferring SOL
    pub recipient: SystemAccount<'info>,
    
    pub system_program: Program<'info, System>,
}
```

### Anchor CPI Approach #2: More Generic CPI

```rust
pub fn sol_transfer(ctx: Context<SolTransfer>, amount: u64) -> Result<()> {
    let from_pubkey = ctx.accounts.sender.to_account_info();
    let to_pubkey = ctx.accounts.recipient.to_account_info();
    let program_id = ctx.accounts.system_program.to_account_info();
    
    // Prepare instruction AccountMetas
    let account_metas = vec![
        AccountMeta::new(from_pubkey.key(), true),  // Sender (mut, signer)
        AccountMeta::new(to_pubkey.key(), false),   // Recipient (mut)
    ];
    
    // SOL transfer instruction discriminator
    let instruction_discriminator = 2u32;
    
    // Prepare instruction data
    let mut instruction_data = Vec::with_capacity(4 + 8);
    instruction_data.extend_from_slice(&instruction_discriminator.to_le_bytes());
    instruction_data.extend_from_slice(&amount.to_le_bytes());
    
    // Create instruction
    let instruction = Instruction {
        program_id: program_id.key(),
        accounts: account_metas,
        data: instruction_data,
    };
    
    // Invoke CPI
    invoke(&instruction, &[from_pubkey, to_pubkey, program_id])?;
    
    Ok(())
}
```

## 6. Creating Accounts in Anchor

### The `#[account(init)]` Constraint

Anchor provides a powerful `init` constraint that handles account creation via CPI automatically.

```rust
#[derive(Accounts)]
pub struct Initialize<'info> {
    #[account(
        init,                    // Triggers CPI to System Program
        payer = user,            // Who pays for account creation
        space = 8 + 4            // 8 discriminator + 4 for u32
    )]
    pub new_account: Account<'info, NewAccount>,
    
    #[account(mut)]
    pub user: Signer<'info>,
    
    pub system_program: Program<'info, System>,
}

#[account]
pub struct NewAccount {
    pub data: u32,  // 4 bytes
}
```

**What `init` does automatically:**
1. Calls System Program's `create_account` via CPI
2. Transfers lamports from `payer` for rent exemption
3. Allocates `space` bytes
4. Sets owner to current program
5. Initializes account discriminator (8 bytes)

## 7. CPI to Create Account in Anchor

### Manual Account Creation (Anchor)

```rust
#[derive(Accounts)]
pub struct Initialize<'info> {
    #[account(init, payer = signer, space = 8 + 4)]
    pub new_account: Account<'info, NewAccount>,
    
    #[account(mut)]
    pub signer: Signer<'info>,
    
    pub system_program: Program<'info, System>,
}
```

This is equivalent to:

```rust
// What Anchor does under the hood
pub fn initialize(ctx: Context<Initialize>) -> Result<()> {
    // Anchor automatically:
    // 1. Creates CPI to System Program
    // 2. Calls create_account instruction
    // 3. Sets owner to this program
    // 4. Allocates space
    // 5. Makes account rent-exempt
    
    // You just initialize your data:
    ctx.accounts.new_account.data = 1;
    
    Ok(())
}
```

## 8. Testing CPIs

### LiteSVM Test Example

```typescript
import * as path from "path";
import {
  Keypair,
  LAMPORTS_PER_SOL,
  PublicKey,
  SystemProgram,
  Transaction,
  TransactionInstruction
} from "@solana/web3.js";
import { LiteSVM } from "litesvm";
import { describe, beforeAll, test, expect } from "bun:test";

describe("Counter Program with CPI Tests", () => {
    let svm: LiteSVM;
    let programId: PublicKey;
    let dataAccount: Keypair;
    let userAccount: Keypair;
  
    const programPath = path.join(import.meta.dir, "counter.so");
  
    beforeAll(() => {
      svm = new LiteSVM();
      programId = PublicKey.unique();
      svm.addProgramFromFile(programId, programPath);
      
      dataAccount = new Keypair();
      userAccount = new Keypair();
  
      svm.airdrop(userAccount.publicKey, BigInt(LAMPORTS_PER_SOL));
    });

    test("Initialize counter using CPI", () => {
        // Build instruction for Initialize
        const instruction = new TransactionInstruction({
            programId,
            keys: [
                { pubkey: dataAccount.publicKey, isSigner: true, isWritable: true },
                { pubkey: userAccount.publicKey, isSigner: true, isWritable: true },
                { pubkey: SystemProgram.programId, isSigner: false, isWritable: false },
            ],
            data: Buffer.from([0]) // Initialize instruction
        });

        const transaction = new Transaction().add(instruction);
        transaction.recentBlockhash = svm.latestBlockhash();
        transaction.feePayer = userAccount.publicKey;
        transaction.sign(dataAccount, userAccount);
        
        let signature = svm.sendTransaction(transaction);
        console.log("Initialize tx:", signature);

        // Verify account created via CPI
        const account = svm.getAccount(dataAccount.publicKey);
        expect(account).toBeTruthy();
        expect(account!.owner).toEqual(programId);
        
        // Verify data initialized
        expect(account!.data[0]).toBe(1);
    });

    test("Double counter", () => {
        const instruction = new TransactionInstruction({
            programId,
            keys: [
                { pubkey: dataAccount.publicKey, isSigner: true, isWritable: true }
            ],
            data: Buffer.from([1]) // Double instruction
        });

        const transaction = new Transaction().add(instruction);
        transaction.recentBlockhash = svm.latestBlockhash();
        transaction.feePayer = userAccount.publicKey;
        transaction.sign(dataAccount, userAccount);
        
        svm.sendTransaction(transaction);

        const account = svm.getAccount(dataAccount.publicKey);
        expect(account!.data[0]).toBe(2);
    });
});
```

## 9. CPI Best Practices

### Security Considerations

| Practice | Description |
|----------|-------------|
| **Validate Program IDs** | Always verify you're calling the correct program |
| **Check Account Ownership** | Ensure accounts are owned by expected programs |
| **Verify Signers** | Confirm required signatures are present |
| **Use PDAs Carefully** | Understand authority when using Program Derived Addresses |
| **Handle Errors** | Properly catch and handle CPI errors |

### Common Patterns

**Pattern 1: SOL Transfer**
```rust
transfer(
    CpiContext::new(
        ctx.accounts.system_program.to_account_info(),
        Transfer {
            from: ctx.accounts.from.to_account_info(),
            to: ctx.accounts.to.to_account_info(),
        },
    ),
    amount,
)?;
```

**Pattern 2: Account Creation**
```rust
#[account(init, payer = user, space = 8 + size)]
pub new_account: Account<'info, MyAccount>,
```

**Pattern 3: Token Transfer (SPL Token)**
```rust
token::transfer(
    CpiContext::new(
        ctx.accounts.token_program.to_account_info(),
        token::Transfer {
            from: ctx.accounts.from,
            to: ctx.accounts.to,
            authority: ctx.accounts.authority,
        },
    ),
    amount,
)?;
```

## Summary

Cross-Program Invocations (CPIs) are fundamental to Solana composability:

1. **CPIs enable composability**: Programs can call other programs
2. **System Program is common target**: Account creation, SOL transfers
3. **Anchor simplifies CPIs**: Helper functions and macros reduce boilerplate
4. **Security is critical**: Always validate program IDs and account ownership
5. **init constraint powerful**: Automatically handles account creation via CPI
6. **Testing essential**: Use LiteSVM or Anchor tests to verify CPI behavior

**CPIs transform Solana from isolated programs into an interconnected ecosystem**—enabling DeFi protocols, NFT marketplaces, and complex dApps to build on each other.

## Key Concepts Table

| Concept | Description |
|---------|-------------|
| CPI | Cross-Program Invocation - calling one program from another |
| invoke() | Native Solana function to execute CPIs |
| CpiContext | Anchor wrapper for CPI context |
| System Program | Built-in program for account creation, transfers |
| init Constraint | Anchor macro that auto-creates accounts via CPI |
| payer | Account paying rent for new account creation |
| space | Bytes to allocate for new account |
| Rent Exemption | Minimum SOL balance to keep account alive |
| Account Ownership | Which program owns/controls an account |
| Program-Derived Address | Deterministic address derived from program |
| Transfer | Moving SOL or tokens between accounts |
| AccountMeta | Metadata describing account (mut, signer, etc.) |
| Instruction Data | Serialized parameters for CPI instruction |
| Discriminator | First 8 bytes identifying account type (Anchor) |
