## 2. BonkBot

**BonkBot manages private keys** by storing them securely on their servers, so users don't have to handle them. They use strong security measures like encryption, secure storage (e.g., hardware security modules), and strict access controls to keep the keys safe. The bot automates transactions by using the private keys in the background, signing actions like sending tokens or interacting with the blockchain without user input. This makes the process seamless and secure for users.

BonkBot solves the problem of making blockchain interactions (like sending tokens or staking) **automatic and easy**, without requiring users to manually handle private keys or sign transactions every time.

### What Does BonkBot Do?

1. **Monitors Transactions**:
   - Watches blockchain activity, like when someone sends tokens to a specific wallet

2. **Automates Actions**:
   - Automatically processes tasks, like sending back rewards or performing actions based on pre-set rules

3. **Handles Private Keys Securely**:
   - Stores private keys securely on its servers and uses them to sign transactions in the background

### How Does BonkBot Solve the Problem?

- **No User Interaction Needed**: Users don't have to manage private keys or manually sign every transaction
- **Secure Automation**: Private keys are kept safe using encryption and strict controls, and BonkBot automates blockchain tasks seamlessly

### How BonkBot Stores Private Keys

BonkBot likely stores private keys in a **secure, encrypted form** to ensure they are protected from unauthorized access. The private keys are most likely stored in one of the following methods:

1. **Encrypted File Storage**:
   - The private key is encrypted using strong encryption algorithms (like **AES-256**) and stored in a secure location on their server

2. **Hardware Security Module (HSM)**:
   - The private key may be stored in a **hardware security module**, which is a physical device designed to securely store and manage cryptographic keys

3. **Environment Variables**:
   - The private key could be stored in **environment variables** on the server, but only accessible by the application with secure access controls

4. **Key Management Systems (KMS)**:
   - A dedicated **key management system** might be used to securely store and manage keys, ensuring that only authorized processes can access them

In all cases, BonkBot ensures that the private key is **never exposed** or accessible to unauthorized users, and it is used for signing transactions automatically in the background, keeping it secure.

### Private Key Security Comparison

| Storage Method | Security Level | Use Case |
|---|---|---|
| Plain text file | ❌ Very Low | Never use in production |
| Environment variable | ⚠️ Low-Medium | Simple apps, careful access control needed |
| Encrypted file (AES-256) | ✅ Good | Most backend services |
| Hardware Security Module (HSM) | ✅✅ Very High | Exchanges, high-value systems |
| Key Management System (KMS) | ✅✅ Very High | AWS KMS, Google Cloud KMS, HashiCorp Vault |

## 3. Project To Build — Cloud Wallet Service

We need to create a website where users can come, sign up and create a **cloud wallet** for themselves. They can login via username and password, create new wallets, ask the backend to sign a transaction on their behalf.

### Architecture

```
Frontend (React)
    │
    ▼
Backend (Node.js / Express)
    │
    ├── Auth layer (JWT)
    │     ├── POST /api/v1/signup
    │     └── POST /api/v1/signin
    │
    ├── Wallet layer
    │     └── Creates + stores Solana keypair per user
    │         (private key encrypted + stored in DB)
    │
    └── Transaction layer
          ├── POST /api/v1/txn/sign
          └── GET  /api/v1/txn?id=<id>
```

### API Specification

#### User Signs Up
```
POST /api/v1/signup

Request Body:
{
  "username": "harkirat",
  "password": "123456"
}

Response 200:
{
  "publicKey": "<solana_public_key>"
}

Response 400:
{
  "message": "User already exists"
}
```

**What happens on signup:**
- Validate username/password
- Generate a new Solana keypair (Keypair.generate())
- Encrypt the private key (AES-256)
- Store encrypted private key + public key in DB
- Return the public key to the user

#### User Signs In
```
POST /api/v1/signin

Request Body:
{
  "username": "harkirat",
  "password": "123456"
}

Response 200:
{
  "jwt": "<jwt_token>"
}

Response 401:
{
  "message": "Incorrect credentials"
}
```

#### User Asks to Sign a Transaction
```
POST /api/v1/txn/sign

Headers:
  Authorization: Bearer <jwt>

Request Body:
{
  "message": "transaction_signature",
  "retry?": true/false
}

Response 200:
{
  "id": "id_to_track_txn"
}
```

**What happens on sign txn:**
- Verify JWT → get user's encrypted private key from DB
- Decrypt private key
- Deserialize the transaction from the message
- Sign the transaction with the keypair
- Broadcast to Solana network
- Store tx ID + status in DB
- Return tracking ID

#### User Asks for the Status of a Transaction
```
GET /api/v1/txn?id=<id>

Response 200:
{
  "signatures": ["xyz"],
  "status": "processing" | "success" | "failed"
}
```
