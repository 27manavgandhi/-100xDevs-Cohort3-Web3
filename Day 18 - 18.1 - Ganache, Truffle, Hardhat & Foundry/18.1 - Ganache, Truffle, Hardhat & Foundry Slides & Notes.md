## 1. Local Development in ETH

Until now, we've used **Remix** for developing our contracts (https://remix.ethereum.org/). While this works, as your project grows and you work in a team, the contract needs to be:

1. **Maintained by various users** — Version control and collaboration
2. **Tested and built** — Comprehensive test suites
3. **Deployed periodically** — CI/CD pipelines

For this, you can use **local development tools** that let you start the blockchain locally and deploy contracts on it.

### Why Local Development?

| Benefit | Description |
|---------|-------------|
| **Fast** | No waiting for block confirmation (instant mining) |
| **Free** | No gas costs for testing |
| **Isolated** | Test without affecting real networks |
| **Debuggable** | Better error messages and stack traces |
| **Repeatable** | Reset state anytime for consistent tests |

## 2. Ganache

**Ganache** is used for starting/creating an Ethereum blockchain locally for testing.

It is similar to **Solana test validator** that lets you start Solana locally.

### Key Features

- Starts a local Ethereum blockchain
- Provides test accounts with ETH
- Instant mining (transactions confirmed immediately)
- GUI and CLI versions available
- Good for beginners

### Installation

```bash
npm install -g ganache
```

### Running Ganache

```bash
ganache
```

This starts a local blockchain at `http://127.0.0.1:8545` with 10 test accounts, each having 100 ETH.

## 3. Truffle

A **world-class development environment**, **testing framework**, and **asset pipeline** for blockchains using the Ethereum Virtual Machine (EVM), aiming to make life as a developer easier.

### Key Features

| Feature | Description |
|---------|-------------|
| **Development** | Write contracts in Solidity, test framework |
| **Testing** | Automated testing with Mocha and Chai |
| **Deployment** | Migration system for deploying to any network |
| **Asset Pipeline** | Built-in support for compiling assets |

### Note

For now, we don't use **Ganache** & **Truffle** anymore.

Instead, we use **Hardhat** and **Foundry**.

## 4. Hardhat

**Hardhat** is a development environment for Ethereum that helps you write, test, and deploy smart contracts.

### Why It's Useful Over Ganache and Truffle

✅ **It allows you to run a local Ethereum blockchain** (like Ganache) and also provides debugging tools

✅ **You can write smart contracts and test them easily** using JavaScript (JS) or TypeScript (TS)

### Contract and Test in JS or TS

- **Smart Contracts**: Written in Solidity, but the testing and interaction with contracts are done in JavaScript or TypeScript
- **Testing**: You write test scripts in JS or TS to interact with your deployed contracts, check for correct behavior, and simulate transactions

### Installation

```bash
npm install --save-dev hardhat
npx hardhat
```

### Project Structure

```
hardhat-project/
├── contracts/          # Solidity contracts
├── scripts/            # Deployment scripts
├── test/              # Test files (JS/TS)
├── hardhat.config.js  # Configuration
└── package.json
```

## 5. Foundry

**Foundry** is another tool used for smart contract development, testing, and deployment. It's known for being **fast** and **efficient**, especially for testing smart contracts.

### It Has 4 Parts

1. **Forge**: Ethereum testing framework (like Truffle, Hardhat)
2. **Cast**: Swiss army knife for interacting with EVM smart contracts, sending transactions, and getting chain data
3. **Anvil**: Local Ethereum node, akin to Ganache, Hardhat Network
4. **Chisel**: Fast, utilitarian, and verbose solidity REPL

### Comparison Table

| Tool | Solana Equivalent | Purpose |
|------|-------------------|---------|
| **Forge** | Anchor | Testing framework |
| **Cast** | spl-token cli | Interact with contracts |
| **Anvil** | solana-test-validator | Local blockchain node |
| **Chisel** | - | Solidity REPL |

### Why It's Useful

✅ **Foundry is very fast**, making it ideal for developers who want to quickly iterate on their contracts and tests

✅ **It provides a suite of tools** for compiling, deploying, and testing contracts, along with a built-in test framework

### Contract and Test in Solidity

- **Foundry uses Solidity** for writing smart contracts, and the testing framework is also built to work directly with **Solidity**
- **It has a simple syntax** for testing, allowing developers to write tests in Solidity rather than JavaScript/TypeScript

### How It Works

**Foundry uses:**
- **Forge** for compiling and testing smart contracts
- **Cast** for interacting with contracts and sending transactions

In short, **Foundry** is a fast, Solidity-native development tool that makes writing, testing, and deploying smart contracts efficient, with a focus on speed and simplicity.

## 7. Emits in Solidity (Events)

In Solidity, **emits** refers to the action of triggering an **event**. Events allow a smart contract to log important data, which can be observed externally (e.g., by front-end apps or other services).

### Real-Life Example

Imagine you're sending money to a friend using an app. After you send the money, the app might show a message like:

**"You sent 100 USD to John."**

This message is an **event** because it's important information that users or the app itself may want to track.

### Why Use Events?

| Reason | Description |
|--------|-------------|
| **Logging** | Record important actions (transfers, approvals) |
| **Off-chain tracking** | Frontend apps can listen to events |
| **Gas efficient** | Cheaper than storing data in contract storage |
| **Searchable** | Indexed parameters allow efficient filtering |

---

## 8. How Events Work in Solidity

### Step 1: Declaring an Event

Before using an event, we first need to **define it** in the contract. This is done with the `event` keyword.

```solidity
event Transfer(address indexed from, address indexed to, uint256 value);
```

**Parameters:**
- `Transfer`: Name of the event
- `from`: The address sending the tokens
- `to`: The address receiving the tokens
- `value`: The amount being transferred

### Step 2: Emitting an Event

To actually **emit** the event (trigger it), we use the `emit` keyword:

```solidity
emit Transfer(msg.sender, recipient, 100);
```

**What happens:**
- `msg.sender`: The address calling the function (the sender)
- `recipient`: The address that will receive the tokens
- `100`: The amount being transferred

---

## 9. Complete Example

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

contract MyToken {
    mapping(address => uint256) public balances;
    
    // Declare the event
    event Transfer(address indexed from, address indexed to, uint256 value);
    
    function transfer(address to, uint256 amount) public {
        require(balances[msg.sender] >= amount, "Insufficient balance");
        
        balances[msg.sender] -= amount;
        balances[to] += amount;
        
        // Emit the event
        emit Transfer(msg.sender, to, amount);
    }
}
```

## 10. Storing Events on the Blockchain

When an event is emitted, it's logged in the blockchain as an **Event Log**.

### Event Log Structure

**Topics**: These are the **indexed parameters** that are used for quick search or filtering. They include:
- `from`: `0x123456789ABCDEF...` (the sender's address)
- `to`: `0xFEDCBA9876543...` (the recipient's address)

**Data**: This is the actual data associated with the event, like the value transferred:
- `value`: `100` (the number of tokens transferred)

### Indexed Parameters in Events

**Indexed** parameters (like `from` and `to`) are special because they are **searchable** and can be used to quickly find specific events. Think of them like **keywords** or **tags** for events.

In our example, the `from` and `to` addresses are indexed, so you can quickly search for all transactions involving a specific address.

### Example Search

You could search for all transactions where someone sends tokens from address `0x123...`:

```javascript
// Search for all Transfer events from specific address
const events = await contract.queryFilter(
  contract.filters.Transfer("0x123...", null, null)
);
```

## 11. Testing Emits

### Foundry Test Example

```solidity
// SPDX-License-Identifier: UNLICENSED
pragma solidity ^0.8.13;

import {Test, console} from "forge-std/Test.sol";
import {MyToken} from "../src/MyToken.sol";

contract CounterTest is Test {
    MyToken public token;
    
    function setUp() public {
        token = new MyToken();
    }
    
    function test_ExpectEmit() public {
        // Check that the following emitted event is identical to the following emitted event
        vm.expectEmit(true, true, false, true);
        
        // The event we expect
        emit Transfer(address(this), 0x075c299EF8dFFF7C9804FE3d0C7963727c0dF5f9, 100);
        
        // The actual transaction
        token.transfer(0x075c299EF8dFFF7C9804FE3d0C7963727c0dF5f9, 100);
    }
}
```

### vm.expectEmit Parameters

```solidity
vm.expectEmit(true, true, false, true);
```

**Parameters (in order):**
1. **Check topic 1** (true/false) - Usually the `from` address
2. **Check topic 2** (true/false) - Usually the `to` address  
3. **Check topic 3** (true/false) - Additional indexed parameter
4. **Check data** (true/false) - The `value` or other non-indexed data

In this example:
- Checking `from` address: `true`
- Checking `to` address: `true`
- Checking topic 3: `false` (not used)
- Checking `value`: `true`

## 13. Std Libraries

Foundry provides several **standard library** functions for testing. These are **cheat codes** that help manipulate the test environment.

### 1. vm.hoax

The `hoax` function is used to **prank the blockchain** by making a transaction seem like it's coming from a different address. It temporarily changes the sender of a transaction for one operation only.

#### How it works

It makes it appear as if a specific address (which you specify) is sending the transaction instead of the default sender (usually `msg.sender`).

#### Example

```solidity
vm.hoax(address, value);
```

- **address**: The address that you want to pretend is sending the transaction
- **value**: The amount of ETH to be sent from that address

#### Real-Life Analogy

Imagine you are at a store, and you want to pay using someone else's credit card for a specific transaction. Once you complete the purchase, you go back to using your own card for the next transactions.

---

### 2. vm.deal

The `deal` function is used to **set the balance** of an address in your test environment. It helps simulate different scenarios by assigning a specific amount of ETH or tokens to an address.

#### How it works

You can directly set the balance of any address to a desired amount in the test.

#### Example

```solidity
vm.deal(account, balance);
```

- **account**: The address whose balance you want to set
- **balance**: The amount of ETH to set for that address

#### Real-Life Analogy

It's like giving someone a **gift card with a preloaded balance** before they go shopping. You can make sure they have enough balance before they make a purchase.

---

### 3. vm.prank, vm.startPrank, and vm.stopPrank

These functions are used to **impersonate another address** for a transaction or multiple transactions.

#### vm.prank(address)

**Temporarily impersonates** another address for **just one transaction**

```solidity
vm.prank(address);
```

#### Example

```solidity
vm.prank(address);
// Borrow a friend's ID for one store visit
```

#### vm.startPrank(address)

**Starts impersonating** an address for **all subsequent transactions**

```solidity
vm.startPrank(address);
```

#### Example

```solidity
vm.startPrank(address);
// Borrow someone's ID and continue to use it for multiple transactions
```

#### vm.stopPrank()

**Ends impersonation**, meaning you go back to using your own address for the following transactions

```solidity
vm.stopPrank();
```

#### Example

```solidity
vm.stopPrank();
// You stop using your friend's ID and go back to using your own
```

### vm.prank(address)

### vm.startPrank(address)

You borrow someone's ID and **continue to use it** for the stores you visit that day

### vm.stopPrank()

You **stop using your friend's ID** and go back to using your own
