## 1. Blockchain Recap

A blockchain is a **decentralized, distributed ledger technology** that records transactions across multiple computers in such a way that the records cannot be altered retroactively without gaining access to more than 51% of the network.

Think of a blockchain as a **digital notebook** where you and a bunch of people can write things down. But here's the twist: once you write something, it can't be erased or changed. It's permanent.

### How does it work?

1. **Blocks and Chains**: Every new piece of information is a "block" that gets added to a long "chain" of other blocks — this creates a "blockchain"
2. **Verification by Many**: Before adding a block, everyone in the group (the network) has to agree that the information in the block is correct — like a group of friends checking that what's written is true before putting it in the notebook
3. **Decentralization**: There's no single person or company in charge of the notebook. Instead, everyone has a copy of it, so no one person can change it secretly
4. **Security**: Each block is locked with a code that links it to the previous one, so it's incredibly hard to change any block without everyone noticing

### Real Life Example

Imagine a group of friends keeping track of who owes whom money:
- Each time someone borrows money, they write it down in the group notebook (adding a block)
- Before writing, everyone double-checks the amount and who borrowed it (verification)
- Once agreed, it's added to the notebook, and no one can change it later
- Everyone has their own copy of this notebook (decentralization), so if someone tries to change their debt, others can compare and see the original record

> That's blockchain! It's a secure, transparent way to keep information that anyone can trust, especially when people don't know each other or fully trust one another yet need a reliable record.

## 2. World State in Ethereum

In simple terms, the **World State** in Ethereum is like a **giant list** that keeps track of the current situation for every account on the network. This includes two types of accounts:

1. **Externally Owned Accounts (EOAs)**: Accounts that people control with their private keys — like personal wallets
2. **Contract Accounts**: Accounts controlled by smart contracts instead of a private key

```
WORLD STATE
┌─────────────┬──────────────────────────────────────────────┐
│ block #1    │ TXN → WORLD STATE #1                         │
│ block #2    │ TXN → WORLD STATE #2                         │
│ block #3    │ TXN → WORLD STATE #3                         │
│ block #4    │ TXN → WORLD STATE #4 → WORLD STATE #5        │
└─────────────┴──────────────────────────────────────────────┘
```

### So, What Does the World State Do?

The World State maps each account's **address** (the unique identifier for each account) to its **current state**. Each account's state includes details like:
- **Balance**: How much ETH the account holds
- **Storage**: Any data stored in the account (especially for contract accounts)
- **Code**: For contract accounts, this includes the actual code that the smart contract runs

### Simple Analogy

Think of the World State as a **spreadsheet** where each row represents an account, and each column holds the current details about that account. Whenever a transaction happens, this spreadsheet updates with the new balances, data, and any other changes.

```
World State (Spreadsheet):
┌───────────────┬─────────┐
│   ACCOUNT 1   │ 1.1 ETH │
│   ACCOUNT 2   │  2 ETH  │
│   ACCOUNT 3   │ 0.5 ETH │
│   ACCOUNT 4   │  2 ETH  │
└───────────────┴─────────┘
```

So, the World State is the complete picture of what every account on Ethereum currently "looks like" at any given moment.

### ETH as a State Machine

Ethereum is a **state machine** whose state changes as more blocks are added to the blockchain. This means that Ethereum constantly updates its **World State** (the current situation of all accounts) whenever a new block of transactions is added.

- Each block = a set of transactions
- Each transaction = a state transition
- New block added → World State updates → new snapshot of all accounts

## 3. Account Types in ETH (EOA vs Contract Accounts)

### Externally Owned Accounts (EOAs)

EOAs are like personal wallets on the Ethereum network.

#### Think of an EOA Like a Personal Bank Account

Imagine you have a bank account that you control with a **PIN code**. This account lets you do things like:
- **Send money** to others
- **Receive money** from others
- **Check your balance** whenever you want

In Ethereum:
- The **bank account** is like an EOA — a simple account where you can store and manage your money (ETH in this case)
- The **PIN code** is like your private key — a secret code only you know that lets you control the account

#### Key Characteristics of EOAs

1. **Controlled by Private Keys (Your Secret PIN)**:
   - Just like you need your PIN to access your bank account, in Ethereum, you need your private key to access your EOA
   - Only you have the key, so only you can send ETH or interact with other accounts from that EOA

2. **No Extra Features (Just Stores and Sends Money)**:
   - An EOA is like a very basic bank account — you can send or receive money, but you can't do any fancy stuff like automate payments or set conditions on the funds
   - In Ethereum, an EOA has no code or special instructions attached to it — it simply holds ETH and sends it when you tell it to

3. **Initiates Transactions (You Start All Actions)**:
   - In Ethereum, **EOAs are the only accounts that can start transactions** (like sending ETH or interacting with smart contracts)
   - Smart contracts can't start actions on their own — they only respond when an EOA interacts with them

4. **Pays a Fee for Each Transaction (Gas Fees)**:
   - Just like you might pay a small fee to transfer money, EOAs have to pay a **gas fee** each time they send a transaction in Ethereum
   - This gas fee is like a tip to the network for processing the transaction

> In Ethereum, an EOA is like a personal bank account with a PIN code that you control. You can send ETH to others, receive ETH, and interact with smart contracts, but you're the one who starts all actions.

### Contract Account

In Ethereum, **Contract Accounts** are special accounts that are governed by code (smart contracts) instead of being controlled by private keys like EOAs.

#### Key Characteristics

1. **Controlled by Code (Smart Contracts)**:
   - Contract Accounts are managed by the smart contract code they contain
   - They don't have private keys and cannot be directly accessed by a person
   - They operate based on the code that was set up when they were created

2. **Cannot Initiate Transactions**:
   - Contract Accounts **can't start transactions on their own**
   - They can only **respond** to incoming transactions from EOAs or other contracts
   - An EOA needs to interact with the Contract Account first to "wake it up"
   - Once they receive a transaction, they can execute functions, update internal data, or even interact with other contracts if programmed to do so

3. **Contains Smart Contract Code**:
   - The smart contract code inside a Contract Account can perform complex tasks
   - This code can contain rules for various financial applications, manage business logic, or execute complex computations
   - The code is usually written in **Solidity** (a programming language for Ethereum) and deployed to the blockchain, where it becomes permanent

4. **Ether Balance**:
   - Like EOAs, Contract Accounts can hold ETH
   - However, they can only spend or transfer it based on the logic written in the smart contract
   - For example, a contract might only release ETH if specific conditions are met

5. **Permanent Deployment**:
   - Once a smart contract is deployed to the Ethereum blockchain, it's **immutable** — it can't be modified or deleted
   - This ensures that the contract's rules remain consistent, but it also means that any mistakes in the code are permanent

> Think of a Contract Account as a **"robot"** that operates according to a specific set of instructions (its code) and can only take actions when someone interacts with it.

## 4. World State — Field by Field

```
World State:
┌────────────────────────────────────────────────────────┐
│ Address (EOA)      → nonce, balance                    │
│                                                        │
│ Address (Contract) → nonce, balance, storage hash,     │
│                      code hash ──→ Storage, Code       │
└────────────────────────────────────────────────────────┘
```

### Nonce

Think of a nonce like a **unique ticket number at a concert**:

**Externally Owned Account (EOA):**
- Each time you buy a ticket (send a transaction), you get a new number (1, 2, 3...)
- This number helps the concert organizers (network) keep track of how many tickets you've bought
- It ensures that no one can use the same ticket twice — prevents double-spending and replay attacks

**Contract Account (Smart Contract):**
- Each time a new game/contract is created, it gets a unique name like "Game1", "Game2"
- This ensures every contract has a unique address and prevents confusion about which contract is which

| Aspect | EOA | Contract Account |
|---|---|---|
| Definition of Nonce | Number of transactions sent by the EOA | Number of contracts created by the account |
| Purpose | Ensures unique transaction ordering | Ensures unique contract addresses |
| Incrementing | Increases after each transaction | Increases after each contract creation |
| Used For | Preventing double-spending, replay attacks, transaction order | Determining the address of newly created contracts |

### Balance

| Feature | EOA | Contract Account (Smart Contract) |
|---|---|---|
| Balance | Holds and manages Ether directly | Holds and manages Ether based on contract code |
| Control | Controlled by a private key | Controlled by contract code |
| Can Send Ether | Yes, via transactions | Yes, but only based on logic in the code |
| Can Receive Ether | Yes | Yes |
| Pays Gas Fees | Yes, from its balance | No (the gas fees are paid by the EOA that called the contract) |

### Storage Hash

#### 1. Storage: The Smart Contract's Filing Cabinet

- Think of **storage** as a **filing cabinet** where a smart contract keeps important information
- Each smart contract has its own filing cabinet, separate from others
- Inside this cabinet, the information is organized in a **key-value store**:
  - **Key**: A label on a drawer (e.g., "User Balances")
  - **Value**: The actual content inside the drawer (e.g., Alice: 10 Ether, Bob: 5 Ether)

#### 2. The Storage Hash: The Summary Page of the Cabinet

- Instead of going through each drawer every time, you create a **summary page** that tells you about all the drawers and their contents
- This summary page represents the **storage hash**
- The **storage hash** is a compact version of everything in the filing cabinet — like a single page that summarizes all the key-value pairs
- It helps you quickly know what data is in your filing cabinet without opening every drawer

#### 3. Why is the Storage Hash Important?

- **Efficiency**: To know the overall state of your filing cabinet (contract storage), you just look at the summary page (storage hash) instead of checking every drawer (individual data)
- **Security**: The storage hash is created using cryptographic techniques, which means it's secure. If anyone tries to change the data in the filing cabinet, the summary page would also change — letting you know something is off

### Code Hash

- Smart contracts in Ethereum are written in high-level languages (like Solidity), but they are **compiled into EVM bytecode** for execution on the Ethereum Virtual Machine (EVM)
- The **code hash** is a cryptographic hash (usually SHA3) of the contract's bytecode — it uniquely identifies the code of the contract
- Once a smart contract is deployed on Ethereum, its code hash is **permanently fixed** unless the contract is destroyed

## 5. Physical Machine vs Virtual Machine

| Feature | Physical Machine | Virtual Machine |
|---|---|---|
| What It Is | An actual computer, like a laptop or server | A pretend computer that runs inside a real computer |
| Components | Made of real parts: CPU, memory (RAM), hard drives | No physical parts — uses the real computer's resources |
| How It Runs | Runs an OS directly on its hardware | Runs on top of another program (hypervisor) |
| Access to Resources | Uses all the power and space of the actual machine | Shares the power and space of the real machine with other VMs |
| Independence | Stands alone — problems don't affect others | Can crash or have issues without impacting other VMs |
| Flexibility | Hard to change — upgrading takes time | Easy to create, copy, or move around |
| Maintenance | Needs physical care (cleaning, replacing parts) | Needs only software updates |
| Cost | Usually costs more | Cheaper overall — many VMs can run on one physical machine |

## 6. JVM (Java Virtual Machine)

The **Java Virtual Machine (JVM)** is a program that allows Java applications to run on any computer, regardless of the operating system. This is part of Java's **"Write Once, Run Anywhere" (WORA)** philosophy.

```
Java code
    ↓
Bytecode
    ↓
   JVM
  / | \
Win  Ubuntu  Linux
```

- When you write a Java program, it is converted into **bytecode**, which the JVM understands and executes
- Think of the JVM like a **DVD player** that can play the same DVD on any TV — this makes Java applications portable and easy to use everywhere

## 7. Ethereum Virtual Machine (EVM)

```
Solidity code
      ↓
   Bytecode
      ↓
     EVM
   /  |  \
Win  Ubuntu  Linux
```

The **Ethereum Virtual Machine (EVM)** is like the Java Virtual Machine (JVM), but it has a different focus. Both are virtual machines that let programs run on different systems without needing changes.

### Key Points

1. **Purpose**: The JVM runs Java programs, while the **EVM runs smart contracts** on the Ethereum blockchain. Smart contracts are like self-executing agreements that run automatically when certain conditions are met

2. **Design**: The EVM is built specifically for the Ethereum network and handles tasks like managing cryptocurrency transactions and executing smart contracts. The JVM, on the other hand, is designed for general-purpose applications

3. **Execution**: The JVM executes Java bytecode, while the **EVM executes bytecode created from smart contracts written in languages like Solidity**

> In short, the EVM and JVM are both virtual machines that allow code to run on different platforms, but the EVM is specifically for Ethereum and smart contracts, while the JVM is for Java programs.

## 8. Solidity, ABI, Bytecode and Opcode

**Flow**: `Solidity code → Bytecode → EVM (Windows / Ubuntu / Linux)`

1. **Solidity Code**: Solidity is a high-level programming language, similar to JavaScript or Python, that developers use to write smart contracts for Ethereum. This code specifies the logic and rules of the smart contract

2. **Bytecode**: Once the Solidity code is written, it gets **compiled** into a low-level format called bytecode. The compiler converts the human-readable Solidity code into bytecode — a set of instructions that the EVM can understand and execute

3. **EVM (Ethereum Virtual Machine)**: The bytecode is then deployed to the EVM — a global decentralized computer that runs on many nodes (computers) around the world

4. **Cross-Platform Compatibility**: The EVM is designed to be platform-independent — it can operate on various operating systems like Windows, Ubuntu, and Linux. This ensures that the network remains consistent and decentralized

> Try Remix IDE: https://remix.ethereum.org

### Summary

Solidity is used to write smart contracts, which are compiled into bytecode. This bytecode is then executed by the EVM, a virtual machine that runs on different operating systems, enabling the deployment of decentralized applications (dApps) on the Ethereum blockchain.

### OPCODE

> Take a look: evm.codes

Each part of the bytecode represents an **opcode** — stands for "operation code." Opcodes are the individual instructions that tell the EVM what to do. Examples of opcodes include:

- **Arithmetic** (e.g., addition or subtraction)
- **Data manipulation** (e.g., storing and retrieving data)
- **Control flow** (e.g., jump to different parts of the code based on conditions)

**Execution in the EVM**: When a smart contract is deployed or executed, the EVM reads the bytecode one opcode at a time, following each instruction to carry out the actions coded in the smart contract.

> In simple terms, **opcodes are like commands in a recipe** for the EVM, telling it exactly what steps to take, one at a time, to carry out the smart contract's functions.

### ABI (Application Binary Interface)

In Ethereum, an **ABI (Application Binary Interface)** is like a **"translation guide"** that allows applications and users to interact with smart contracts.

1. **Purpose of ABI**: When a smart contract is deployed on the Ethereum blockchain, it operates using bytecode that the EVM can understand. However, people and external applications need a way to communicate with this contract — to call its functions or read its data. The **ABI defines how this communication happens**

2. **Defining Contract Functions**: The ABI includes details about the contract's functions, specifying:
   - The names of the functions
   - The types of inputs each function requires
   - The types of outputs (if any) it returns

3. **Encoding and Decoding Data**:
   - When an application sends data to the smart contract (calling a function), the ABI tells it how to **encode** this data into the proper format
   - When the contract sends data back (returning a result), the ABI tells the application how to **decode** this data

4. **Practical Example**: Let's say there's a smart contract that has a function to check a user's balance. The ABI would specify that to call the "checkBalance" function, you need to provide the user's address as input, and it will return a number (the balance) as output. An application (such as a wallet) would use the ABI to correctly format the call to "checkBalance" and understand the response

> In simple terms, the **ABI is a map that explains how to interact with a smart contract**, making sure data is sent and received in the right way so applications and users can work with the contract accurately.

## 9. Architecture of EVM (Memory, Storage and Contract)

```
                    EVM
┌───────────────────────────────────────────────────────┐
│  ROM (EVM CODE)                                       │
│  PROGRAM COUNTER    │ STACK │ MEMORY │ ACCOUNT STATE  │
│  GAS AVAILABLE      │       │        │                │
└───────────────────────────────────────────────────────┘
```

The EVM has several key components during contract execution:

- **ROM (EVM Code)**: The bytecode of the smart contract — read-only, never changes during execution
- **Program Counter**: Tracks which opcode is currently being executed (like a bookmark in the recipe)
- **Gas Available**: How much gas (computational budget) is left for this transaction
- **Stack**: A temporary last-in-first-out data structure used for computations during execution
- **Memory**: Temporary storage during a single function call — cleared after execution
- **Account State**: Access to the blockchain's persistent storage (balances, contract storage)
