## 1. Immutable Contracts

### Real-Life Analogy

Think of deployed contracts like **inscriptions on stone tablets**:
- Once carved, the words cannot be changed
- Everyone can read what's written
- Trust comes from knowing it won't change
- But if there's a mistake, you can't erase it

### The Dark Side of Immutability

**Problem**: If a contract is deployed with a vulnerability, you cannot directly fix it.

**Famous Example**: Akutars Exploit
- $34 million locked in a smart contract forever
- Bug in the contract made funds unrecoverable
- Reference: https://nftnow.com/features/akutars-exploit-34-million-locked-in-smart-contract/

### Solana Comparison

> рҹ’Ў **Note**: This is in stark contrast with Solana, where contracts are upgradable by default, and can be forced to be `immutable` by updating the `update authority`

## 2. The Need for Upgradability

While immutability is important, there are scenarios where the ability to upgrade a contract is desirable:

| Use Case | Description |
|----------|-------------|
| **Bug Fixes** | Fixing vulnerabilities or logical flaws |
| **New Features** | Adding new functionality |
| **Optimizations** | Improving performance or reducing costs |
| **Regulatory Compliance** | Adapting to new regulations |
| **Protocol Improvements** | Enhancing the protocol over time |

Without upgradability, once deployed, a smart contract is effectively **"set in stone."**

## 3. Naive Approaches to Upgradability

### Approach 1: Parameterize Everything

Make everything `configurable` by default so if there is ever a change you need to make, there is a `backdoor` for you to make it.

This is very similar to how `mobile apps` are made highly `configurable` so they can be updated in the background without needing to go through `App store` / `Play store` approval.

```solidity
// SPDX-License-Identifier: GPL-3.0
pragma solidity ^0.8.22;

contract Storage {
    uint multiplier;
    uint num;

    constructor() {
        num = 0;
        multiplier = 1;
    }

    function setNum(uint _num) public {
        num = _num * multiplier;
    }

    function getnum() public virtual returns (uint) {
        return num;
    }

    function setMultiplier(uint _multiplier) public {
        multiplier = _multiplier;
    }
}
```

**Pros:**
- вң… Easy to implement
- вң… No need to understand proxies

**Cons:**
- вқҢ Limited flexibility
- вқҢ Can't add new functions
- вқҢ Becomes messy with many parameters

### Approach 2: Migrating Contracts

Ask everyone to upgrade to a new smart contract address.

**Pros:**
- вң… Complete control over new contract
- вң… Can fix any issue

**Cons:**
- вқҢ Storage needs to be migrated
- вқҢ Everyone needs to update contract address
- вқҢ Expensive gas costs for migration
- вқҢ Poor user experience

## 4. delegateCall - The Foundation of Upgradability

`delegatecall` is a low-level function used to execute code from another contract while maintaining the context (storage, msg.sender, msg.value) of the calling contract.

### Syntax

```solidity
(bool success, bytes memory returnData) = targetContract.delegatecall(data);
```

### delegateCall vs Call

| Aspect | `call` | `delegatecall` |
|--------|--------|----------------|
| **Code Execution** | Executes in target contract | Executes in target contract |
| **Storage** | Modifies target's storage | Modifies caller's storage |
| **msg.sender** | Becomes contract address | Stays same as original |
| **Use Case** | Interacting with another contract | Upgradable proxies, modular contracts |

### Example Contract

```solidity
// SPDX-License-Identifier: GPL-3.0
pragma solidity >=0.7.0 <0.9.0;

contract Storage {
    uint public num;
    address implementation;

    constructor(address _implementation) {
        num = 0;
        implementation = _implementation;
    }

    function setNum(uint _num) public {
        (bool success, ) = implementation.delegatecall(
            abi.encodeWithSignature("setNum(uint256)", _num)
        );
        require(success, "Error while delegating call");
    }
}

contract Implementation {
    uint public num;

    function setNum(uint _num) public {
        num = _num;
    }
}
```

### Key Point

When `Storage` calls `Implementation.setNum()` via delegatecall:
- **Implementation's code** runs
- **Storage's num variable** gets updated
- **NOT** Implementation's num variable

## 5. The Proxy Pattern

The most common design pattern for upgradable contracts is the **Proxy Pattern**. This involves separating the contract's data (storage) and logic (code).

    uint public num;

    uint public num;

2. Deploy StorageProxy with Implementationv1 address

## 6. Storage Layout - The Critical Concept

When using `delegatecall`, the **implementation contract's code** is executed in the **proxy's storage context**. For this to work properly, the **storage layout** in the proxy must match the **storage layout** in the implementation.

### The Problem with Ownable

```solidity
import "@openzeppelin/contracts/access/Ownable.sol";

contract StorageProxy is Ownable {
    uint public num;           // Slot 1 (NOT Slot 0!)
    address implementation;     // Slot 2
    
    constructor(address _implementation) Ownable(msg.sender) {
        implementation = _implementation;
    }
}
```

**Storage Layout:**
- **Slot 0** вҶ’ `_owner` (from Ownable)
- **Slot 1** вҶ’ `num`
- **Slot 2** вҶ’ `implementation`

But in the implementation:

```solidity
contract Implementationv1 {
    uint public num;  // Slot 0 вҡ пёҸ MISMATCH!
    
    function setNum(uint _num) public {
        num = _num;
    }
}
```

**Storage Layout:**
- **Slot 0** вҶ’ `num`

### The Solution

Match the storage layout by adding placeholder variables:

```solidity
contract Implementationv2 {
    address public _owner;  // Slot 0 (matches proxy)
    uint public num;        // Slot 1 (matches proxy)

    function setNum(uint _num) public {
        num = _num * 2;
    }
}
```

## 7. Fallback Functions

The **fallback function** is executed when the contract receives a call that doesn't match any existing function signature.

### Properties

- No name (hence "fallback")
- Cannot have parameters
- Cannot return anything
- Marked as `external`
- Can be `payable`

### Code Example

```solidity
contract Proxy {
    address public implementation;

    constructor(address _implementation) {
        implementation = _implementation;
    }

    fallback() external payable {
        // Forward ALL calls to the implementation contract
        (bool success, ) = implementation.delegatecall(msg.data);
        require(success, "Delegatecall failed");
    }
}

contract ImplementationV1 {
    uint public num;
    address public implementation;

    function setNum(uint _num) public {
        num = _num;
    }
}
```

### How It Works

```
User calls: proxy.setNum(5)
       вҶ“
No setNum() in proxy
       вҶ“
fallback() is triggered
       вҶ“
delegatecall to implementation
       вҶ“
Implementation's setNum() runs
       вҶ“
Proxy's storage is updated
```

## 8. ERC-1967 - Standard Proxy Storage Slots

Reference: https://eips.ethereum.org/EIPS/eip-1967

ERC-1967 defines **standard storage slots** for proxy contracts to avoid storage collisions.

### The Magic Number

```solidity
bytes32 internal constant IMPLEMENTATION_SLOT = 
    0x360894a13ba1a3210667c828492db98dca3e2076cc3735a920a3ca505d382bbc;
```

This slot is calculated as: `keccak256("eip1967.proxy.implementation") - 1`

### Why This Matters

By storing the implementation address in a **special slot** far from normal storage:
- Avoids collision with implementation contract variables
- Follows a standard that tools understand
- Makes proxies more secure

### Implementation

```solidity
import "@openzeppelin/contracts/utils/StorageSlot.sol";

contract StorageProxy {
    uint public num;
    bytes32 internal constant IMPLEMENTATION_SLOT = 
        0x360894a13ba1a3210667c828492db98dca3e2076cc3735a920a3ca505d382bbc;

    constructor(address _implementation) {
        setImplementation(_implementation);
    }

    fallback() external {
        (bool success, ) = getImplementation().delegatecall(msg.data);
        if (!success) {
            revert();
        }
    }

    function getImplementation() internal view returns (address) {
        return StorageSlot.getAddressSlot(IMPLEMENTATION_SLOT).value;
    }

    function setImplementation(address _implementation) public {
        StorageSlot.getAddressSlot(IMPLEMENTATION_SLOT).value = _implementation;
    }
}
```

## 9. Transparent Proxy Pattern

Reference: https://eips.ethereum.org/EIPS/eip-1967

The **Transparent Proxy** pattern solves the problem of function selector collisions between the proxy and implementation.

### The Problem

What if both proxy and implementation have a function with the same name?

```solidity
// Proxy has upgradeToAndCall()
// Implementation also has upgradeToAndCall()
// Which one gets called? рҹӨ”
```

### The Solution

**Transparent Proxy Rule:**
- If caller is **admin**: Execute proxy functions (upgrade, etc.)
- If caller is **anyone else**: Forward to implementation

### OpenZeppelin Implementation

```solidity
import {TransparentUpgradeableProxy} from 
    "@openzeppelin/contracts/proxy/transparent/TransparentUpgradeableProxy.sol";

contract StorageProxy is TransparentUpgradeableProxy {
    constructor(
        address _logic,
        address initialOwner,
        bytes memory _data
    )
        payable
        TransparentUpgradeableProxy(_logic, initialOwner, _data)
    {}
}

contract ImplementationV1 {
    uint256 public num;

    function setNum(uint256 _num) public {
        num = _num;
    }
}
```

### Deployment Steps

1. Deploy ImplementationV1
2. Deploy StorageProxy with:
   - Logic address (ImplementationV1)
   - Admin address
   - Init data (empty bytes)
3. Interact with proxy as if it's the implementation

### Advantages of UUPS

- **Gas Savings**: Proxy is simpler (no admin checks)
- **Flexibility**: Each implementation controls its upgrade
- **Smaller Proxy**: Less deployment cost

## 11. Common Proxy Patterns Summary

| Pattern | Upgrade Logic | Gas Cost | Complexity | Use Case |
|---------|---------------|----------|------------|----------|
| **Transparent** | In Proxy | Higher | Medium | General purpose, admin-controlled |
| **UUPS** | In Implementation | Lower | Higher | Gas-sensitive, implementation-controlled |
| **Beacon** | Centralized | Medium | Low | Multiple proxies sharing one implementation |

## Summary

Upgradable contracts in Ethereum solve the immutability problem while maintaining security:

1. **Immutability** is important but creates challenges
2. **delegateCall** enables code reuse with storage separation
3. **Proxy Pattern** separates storage from logic
4. **Storage Layout** must match between proxy and implementation
5. **Fallback Functions** route unknown calls to implementation
6. **ERC-1967** standardizes storage slots
7. **Transparent Proxy** prevents function collisions with admin checks
8. **UUPS** puts upgrade logic in implementation for gas savings
9. **Each pattern** has tradeoffs in gas, complexity, and control

The key principle: **Proxy holds storage, Implementation holds logic, delegateCall connects them.**

## Key Concepts Table

| Concept | Description |
|---------|-------------|
| **Immutability** | Deployed contracts cannot be changed |
| **delegateCall** | Execute code in another contract using caller's storage |
| **Proxy Pattern** | Separate storage (proxy) from logic (implementation) |
| **Storage Slot** | Position where variable is stored (Slot 0, 1, 2...) |
| **Storage Layout** | Order and position of variables in contract |
| **Fallback Function** | Called when no function matches the call |
| **Function Selector** | First 4 bytes of function signature hash |
| **ERC-1967** | Standard for proxy storage slots |
| **Transparent Proxy** | Admin-gated upgrades in proxy |
| **UUPS** | Upgrade logic in implementation |
| **Beacon Proxy** | Multiple proxies sharing one implementation |
| **Storage Collision** | When proxy and implementation variables overlap |
| **Initializer** | Replaces constructor in upgradeable contracts |
| **ProxyAdmin** | Contract managing proxy upgrades |
