## 1. Custom Tokens

**Custom tokens** are any tokens other than the native cryptocurrency (like ETH on the Ethereum blockchain). They are built using the blockchain's standards (like ERC-20 or ERC-721 for Ethereum) and represent unique digital assets.

### Key Characteristics

| Characteristic | Description |
|----------------|-------------|
| **Built on blockchain** | Uses existing blockchain infrastructure (Ethereum, Solana, etc.) |
| **Follow standards** | ERC-20 for fungible tokens, ERC-721 for NFTs |
| **Represent digital assets** | Can represent anything: stablecoins, game items, company shares |
| **Smart contract managed** | Balance tracking and transfers handled by contract code |

## 5. Where are Native Balances Stored?

### World State Diagram

```
┌─────────────────────────────────────────────────────────┐
│                     World State                         │
│              (Universal Ledger)                         │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  Address (EOA)                                          │
│    ├─── Nonce                                           │
│    ├─── Balance  ← EOA Balance (Native ETH)            │
│    └─── Code Hash                                       │
│                                                         │
│  Address (Contract)                                     │
│    ├─── Nonce                                           │
│    ├─── Balance                                         │
│    ├─── Storage Hash                                    │
│    │      ├─── Storage (ledger)                         │
│    │      └─── Code Hash                                │
│    └─── Code                                            │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

The diagram shows how data (like balances) is organized and stored in Ethereum's **World State**, which is like a universal ledger.

### Two Main Types of Addresses in Ethereum

#### 1. EOA (Externally Owned Account)
This is a **regular wallet** (like your personal bank account)

**What it stores:**
- **Nonce**: The number of transactions you've made (like your account's transaction history counter)
- **Balance**: How much money (ETH) you have in your account

#### 2. Contract Address
This belongs to a **smart contract** (like a vending machine holding tokens and performing automated actions)

**What it stores:**
- **Nonce**: The number of times the locker has been accessed or used
- **Balance**: How much money or tokens the locker holds
- **Storage Hash**: Details about the locker contents (like items inside it)
- **Code Hash**: The instructions for how the locker operates (e.g., "open when payment is received")

This diagram explains that **custom tokens** (like USDC, DAI, etc.) have their balances stored differently compared to native tokens (like ETH).

### Key Points

**Native Token Balances (ETH)**: Stored directly in the main bank account ledger
- Simple and directly recorded in the bank's ledger

**Custom Tokens**: Stored in a smart contract's "storage"
- Like items stored in lockers managed by the bank (smart contract)

### Explanation

#### 1. EOA (Externally Owned Account)
**This is your wallet address**
- It doesn't directly store custom token balances
- Instead, it interacts with the smart contract to display your token balance

#### 2. Smart Contract Address
**Stores:**
- **Native Balance**: If someone sends ETH to the contract
- **Storage**: The ledger for custom token balances
  - For example, if 10 USDC belongs to your wallet, this is recorded here

**Example:**
For a Decentralized Uber app, you would store things like:
- The "storage" contains details of individual customers and their token balances
- Who owns what: The "storage" contains details of individual customers and their token balances
- Rules for access: The "code" defines how you can deposit, withdraw, or transfer these tokens

## 14. What Should a Token Contract Look Like?

### Basic Structure

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

contract Token {
    string public name = "KiratCoin";
    uint public supply = 0;
    address public owner;
    mapping(address => uint) public balances;
    
    constructor() {
        owner = msg.sender;
    }
    
    function mint(address to, uint amount) public {
        require(msg.sender == owner);
        balances[to] += amount;
        supply += amount;
    }
    
    function transfer(address to, uint amount) public {
        uint balance = balances[msg.sender];
        require(balance >= amount, "You don't have enough balance");
        balances[msg.sender] -= amount;
        balances[to] += amount;
    }
    
    function burn(uint amount) public {
        uint balance = balances[msg.sender];
        require(balance >= amount, "You don't have enough balance");
        balances[msg.sender] -= amount;
        supply -= amount;
    }
}
```

### Key Functions

1. **Defines a token**: The contract creates a token named "KiratCoin"
2. **Keeps track of balances**: Maintains a record of how much each address owns
3. **Allows minting**: The owner can create new tokens and assign them to any address
4. **Allows transferring**: Users can send their tokens to others
5. **Allows burning**: Users can destroy their tokens, reducing the total supply

## 15. Real-Life Example: Company Reward Points System

**Token = Reward Points**: Think of this contract as creating a company's reward points, e.g., "Kirat Points"

### Minting (Creating Points)
- The company (owner) can add new reward points to customers' accounts (e.g., giving 100 points for a purchase)
- Function: `mint(address, amount)` → The company decides how many points each customer gets

### Transferring Points
- Customers can transfer points to others (e.g., gifting points to friends)
- Function: `transfer(address, amount)` → Points move from one customer's account to another

### Burning Points
- Customers can "spend" or remove points when redeemed (e.g., using 50 points for a discount)
- Function: `burn(amount)` → Points are removed from the customer's balance and the total supply

---

## 16. Example Flow

### 1. Minting
- The company adds 500 points to Customer A using `mint(addressA, 500)`
- Now, `balances[addressA] = 500`

### 2. Transfer
- Customer A sends 100 points to Customer B using `transfer(addressB, 100)`
- Updated balances:
  - `balances[addressA] = 400`
  - `balances[addressB] = 100`

### 3. Burn
- Customer A spends 200 points using `burn(200)` (reduces total supply by 200)

This is how tokens work in blockchain systems, similar to managing reward points!

## 17. Allowances/Delegation in Tokens

### What is an Allowance?

**It allows a token holder to give permission** to another address (like a smart contract or person) to **spend a specified amount** of their tokens on their behalf.

### Key Components in KiratCoin

1. **Balances**: Keeps track of how many tokens each address owns
2. **Allowances**: Tracks how many tokens a third party (delegate) is allowed to spend from the token holder's balance

## 20. Raman Uses Allowance

Raman spends 50 tokens on Harkirat's behalf:

**Function**: `transferFrom(address from, address to, uint amount)`

```solidity
function transferFrom(address from, address to, uint amount) public {
    uint balance = balances[from];
    require(balance >= amount, "You don't have enough balance");
    
    uint currentAllowance = allowances[from][msg.sender];
    require(currentAllowance >= amount);
    
    balances[from] -= amount;
    balances[to] += amount;
    allowances[from][msg.sender] -= amount;
}
```

Raman can now spend up to 100 tokens from Harkirat's balance, but no more.

**Updated State:**
- Harkirat's balance = `original - 50`
- Allowance for Raman = `100 - 50`

## 21. Control and Security

**Harkirat can revoke or increase** the allowance anytime:

- **Increase**: `approve(ramanAddress, 200);` (Now Raman can spend 200)
- **Revoke**: `approve(ramanAddress, 0);` (Raman can no longer spend)

## 23. How Should the Contract Change?

```solidity
mapping(address => mapping(address => uint)) public allowances;

function approve(address spender, uint amount) public returns (bool) {
    allowances[msg.sender][spender] = amount;
    return true;
}

function transferFrom(address from, address to, uint amount) public {
    uint balance = balances[from];
    require(balance >= amount, "Insufficient balance");
    
    uint currentAllowance = allowances[from][msg.sender];
    require(currentAllowance >= amount, "Insufficient allowance");
    
    balances[from] -= amount;
    balances[to] += amount;
    allowances[from][msg.sender] -= amount;
}
```
